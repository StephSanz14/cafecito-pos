= MIDDLEWARE ERRORHANDLER = 

Para empezar a programar en backend, creamos un middleware para manejo de errores, lo vamos a llamar errorHandler, este archivo, ayuda a responder
400, 404, 422 su trabajo es atrapar errores, guardarlos en un archivo, y responderle algo al cliente sin romper el código
Solo se ejecuta cuando pasa esto en alguna ruta: throw new Error('Algo salió mal'); o next(error); y en el archivo de server.js lo trabajamos así: app.use(errorHandler);
Express sabe que es un middleeate de error porque tiene 4 parámetros (err, req, res, next)
Imagina que tienes esto GET /products Algo falla (base de datos, lógica, etc.). Entonces Express dice: “Ok, ocurrió un error… ¿tengo un middleware de errores Y encuentra este archivo.
PASO 1 LLEGA EL ERROR
const errorHandler = (err, req, res, next) => {}
Aquí Express te entrega:
err → el error real
req → qué pidió el usuario
res → para responder
next → por si hubiera otro middleware (aquí no lo usas)
PASO 2 SE CREA LA INFO DEL ERROR
const dateTime = new Date();
Guarda cuando paso el error
req.method  → GET / POST / PUT
req.url     → /products
err.message → qué salió mal
err.stack   → dónde ocurrió
PASO 3 DECIDE DONDE GUARDAR EL ERROR
const logFilePath = path.join(__dirname, '../../logs/error.log');
PASO 4 GUARDA EL ERROR EN EL ARCHIVO
PASO 5 RESPONDE AL CLIENTE 
res.status(500).json({
  status: 'error',
  message: 'Internal Server Error'
});

=CONSTRUCCION POR BLOQUES=
Al crear un bloque de trabajo en backend debemos de seguir el siguiente orden
Modulos
Controladores
Rutas

1️⃣ MÓDULOS (Models / Schemas)
¿Qué son?
Son la definición de tus datos.
Aquí dices:
Qué es una venta
Qué campos tiene
Qué es obligatorio
Qué relaciones existen

👉 Ejemplo mental:
“Así luce una venta en mi sistema”

¿Qué hacen?
Validan datos
Definen estructura
Conectan con la base de datos

¿Por qué van primero?

Porque:
Todo lo demás depende de ellos
No puedes guardar, leer o modificar algo que no está definido
🧠 Sin modelos, el backend no sabe qué es una venta.

2️⃣ CONTROLADORES (Controllers)
¿Qué son?

Son la lógica del negocio.
Aquí decides:
Cómo se crea una venta
Qué pasa cuando se vende
Cómo se calculan totales
Qué validaciones extra se hacen

👉 Ejemplo mental:
“Qué debe pasar cuando alguien vende algo”

¿Qué hacen?

Usan los modelos
Procesan datos
Ejecutan reglas del negocio
Responden al cliente (JSON)

¿Por qué van después?

Porque:
Los controladores usan los modelos
Necesitan saber cómo está estructurada la información

🧠 El controlador piensa, el modelo solo define.

3️⃣ RUTAS (Routes)
¿Qué son?

Son las puertas de entrada a tu backend.
Aquí defines:
Qué URL existe
Qué método HTTP se usa
A qué controlador apunta

👉 Ejemplo mental:
“Por dónde entra la petición”

¿Qué hacen?

Reciben requests
Llaman al controlador correcto
No contienen lógica pesada

¿Por qué van al final?

Porque:
Las rutas solo conectan
No tienen sentido sin controladores
No hacen nada por sí solas

🧠 La ruta no piensa, solo dirige.

🧩 El flujo completo (muy importante)
Cliente (Frontend / Postman)
        ↓
      Ruta
        ↓
   Controlador
        ↓
      Modelo
        ↓
   Base de Datos


Y de regreso 👇

Base de Datos
        ↑
      Modelo
        ↑
   Controlador
        ↑
      Ruta
        ↑
     Cliente

🔑 Por qué ESTE orden es una buena práctica

Separación de responsabilidades

Modelo → estructura
Controlador → lógica
Ruta → navegación

Código más limpio
No mezclas consultas con URLs
No repites lógica
Escalabilidad
Agregar endpoints es fácil
Cambiar lógica no rompe rutas
Mantenimiento
Si falla algo, sabes dónde buscar


Regla mental para que nunca te confundas

Modelo define → Controlador decide → Ruta expone
